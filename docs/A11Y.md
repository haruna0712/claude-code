# アクセシビリティ戦略 (A11Y Strategy)

> Version: 0.1 (Draft)
> 最終更新: 2026-04-19
> スコープ: エンジニア特化型 SNS（Next.js 14 + Tailwind + shadcn/ui）
> 準拠目標: **WCAG 2.2 Level AA**

本書は [SPEC.md](./SPEC.md) §17.2「アクセシビリティ」の詳細化ドキュメントである。UI 設計・実装・QA の各工程で参照し、エンジニアだけでなくスクリーンリーダー利用者・ロービジョン・運動機能障害・認知特性差のあるユーザーにも平等な情報と操作を届けることを目的とする。

---

## 1. a11y 方針と目標

### 1.1 基本方針（POUR）

すべての画面・コンポーネントは WCAG 2.2 の 4 原則 **Perceivable / Operable / Understandable / Robust** に沿って設計する。日本語話者のエンジニアが主要ユーザーであることを踏まえ、日本語 NVDA・VoiceOver（macOS/iOS）・TalkBack での読み上げ品質を最優先する。

### 1.2 準拠スコープ

| 対象 | 準拠レベル | 補足 |
|---|---|---|
| 公開ページ（ツイート詳細 / プロフィール / タグ / 掲示板 / 記事） | **AA 必達** | SEO 対象でもあり、未ログイン流入口のため最優先 |
| ログイン必須の主要画面（ホーム TL / 通知 / DM / 検索 / 投稿コンポーザー） | **AA 必達** | Phase 1–4 で実装する中核機能 |
| 設定・管理画面 | AA 努力 | Phase 4 以降で順次対応 |
| Django admin | 対象外 | 上流の Django 標準に準じる |

### 1.3 監査プロセス

1. **自動検査**: axe-core を Playwright E2E に組み込み、PR ごとに公開ページ + 主要画面を走査。重大度 `serious` / `critical` で CI をレッド化（§8）。
2. **Lighthouse a11y**: Lighthouse CI で a11y スコア **95 以上** を閾値とする。
3. **手動キーボード操作**: マウスを使わず Tab/Shift+Tab/Enter/Space/Esc/方向キーのみで全主要フローを完走できることを確認。
4. **スクリーンリーダー手動検証**:
   - macOS VoiceOver + Safari（必須）
   - Windows NVDA + Firefox（必須）
   - iOS VoiceOver + Mobile Safari（Phase 4 以降）
   - Android TalkBack + Chrome（Phase 4 以降）
5. **コントラスト監査**: Figma レビュー時点で Stark プラグインないし `oklch` 直算で 4.5:1 / 3:1 を事前検証。ダークモードは別個に実測する。
6. **実ユーザーテスト**: Phase 10 で視覚障害・運動機能障害のある外部モニター 2 名以上による受け入れを実施。

## 2. 主要コンポーネントの a11y 要件

### 2.1 ツイートカード

- ルート要素は `<article aria-labelledby="tweet-<id>-author" aria-describedby="tweet-<id>-body">`。
- 著者名は `<a>` でプロフィールへ、ハンドル（`@handle`）は視覚的には小さくとも SR では省略しない。
- 添付画像の `alt`: 投稿時に入力された代替テキストを使用。未入力時は「添付画像（代替テキスト未設定）」ではなく **「画像：<投稿者名> のツイート添付画像」** を fallback とし、情報欠落を明示。
- コードブロック: `<pre><code class="language-ts" tabindex="0" aria-label="TypeScript コード">` とし、フォーカス可能にしてスクロール可能領域を SR に伝える。シンタックスハイライトは色のみの差異なので、装飾扱い（`aria-hidden` は付けないが、構造的には `<code>` 内テキストで意味が通ること）。
- RT / 引用 / リプライ / リアクション / ブックマーク / その他メニューのアイコン only ボタンは **`aria-label`** 必須。
  - 例: `aria-label="リプライ（現在 3 件）"` のように **件数を読み上げる**。
  - 状態（押下済み）は `aria-pressed="true"` またはラベルに「解除」を含める。
- Bot 投稿は `<span class="sr-only">Bot アカウント</span>` をカード冒頭に置き、色/絵文字のみに依存しない。
- 記事連携ツイート（`from_article`）は `<span class="sr-only">記事投稿</span>` で明示。

### 2.2 リアクションメニュー（ロングプレス/ホバー）

- 既定の「いいね風」ショートタップ、長押しで 10 種展開、ホバーで展開、の 3 経路に加え **キーボード代替** を必ず用意する。
  - リアクションボタンに Tab でフォーカス → **Enter/Space で短押し**（既定の like）、**Alt+Enter** または「リアクションを選ぶ」補助ボタンで選択メニューを開く。
  - 開いたメニューは `role="menu"` + `role="menuitemradio"` で 10 種を提示し、矢印キーで移動、Enter で決定、Esc でクローズ。
- `aria-haspopup="menu"` / `aria-expanded` を親ボタンに付与。
- 各リアクションは絵文字 + ラベル（`💡 面白い`）の組み合わせで読み上げ、**色だけ・絵文字だけに依存しない**。
- モバイルでは長押し UX に加えて、リアクション右横に小さな **「…」(他のリアクションを選ぶ)** ボタンを 24x24px 以上で常時表示し、スイッチコントロールからもアクセス可能にする。

### 2.3 ツイート投稿コンポーザー

- `<form role="form" aria-label="新しいツイート">`。
- 本文 `<textarea>`: `aria-describedby` で注意書き・文字数カウンタ・バリデーションメッセージを連結。
- **文字数カウント**: カウンタは `aria-live="polite"` + `aria-atomic="true"` の領域。毎打鍵での読み上げを避けるため、**閾値ベース**で更新する（残り 20 字未満、残り 0、超過時）。超過時のみ `role="alert"` 相当の assertive 通知を別途トリガー。
- 画像 alt 入力:
  - **方針決定**: MVP では **「添付画像ごとに alt 入力を強く推奨（必須ではない）」**。必須化するとエンジニア流入の摩擦になる懸念があるが、未入力時は投稿ボタン横に `⚠ 代替テキスト未設定の画像があります` を表示し、`aria-describedby` でボタンに結ぶ。
  - Phase 10 までに「OS の画像説明 API や Claude Vision で自動 alt を提案 → ユーザーが承認」フローを検討する（オープンイシュー扱い）。
- タグチップス（最大 3 個）は `role="listbox"` + `role="option"` のコンボボックスパターン、削除は Backspace で末尾タグを除去し `aria-live="polite"` で通知。
- 投稿ボタンは送信中 `aria-busy="true"` + `aria-disabled="true"`。成功・失敗は `role="status"` / `role="alert"` で領域に反映。

### 2.4 ホームタイムライン（新着挿入）

- TL コンテナは `<main><ol role="feed" aria-busy="…" aria-label="ホームタイムライン">`。各ツイートは `<li role="article" aria-posinset="…" aria-setsize="…">`。
- WebSocket / ポーリングで新着が届いたら、**自動挿入はしない**（SR 利用者とスクリーン位置が大幅にずれるため）。代わりに画面上部に **「新着ツイート <n> 件を表示」ボタン** を出し、`aria-live="polite"` で「新着 3 件のツイートがあります」と通知。
- 「新着を表示」押下で挿入 → フォーカスは一番上の新ツイート先頭に移動し、`aria-live` で「最新のツイートに更新しました」と告知。
- 無限スクロールのローディングは `role="status" aria-live="polite"` に「さらにツイートを読み込み中」を出す。

### 2.5 DM 画面

- 会話一覧は `role="list"`、各項目は `role="listitem"` でリンク化。未読は `aria-label` に「未読 2 件」を含める。
- メッセージスレッドは `<ol aria-live="polite" aria-relevant="additions" aria-atomic="false">`。自分と相手の新規発話のみ追加通知し、既読更新・タイピングは通知しない（`aria-relevant="additions"` で抑止）。
- タイピング中表示（「相手が入力中...」）は **視覚のみ**、`aria-hidden="true"` としてスクリーンリーダーを鳴らし続けない。ただし 3 秒以上継続した場合に限り 1 回だけ `role="status"` で知らせる（うるささを抑える）。
- 添付画像は `alt` 必須（プレースホルダ: `<送信者名>が送信した画像`）、ファイルは `role="link"` + `aria-describedby` でサイズ・種類を読み上げ。
- メッセージ入力欄は `<textarea aria-label="メッセージを入力" aria-multiline="true">`、送信は Enter（改行は Shift+Enter）、切り替えは設定で変更可能（運動機能・認知アクセシビリティ配慮）。

### 2.6 通知ベル

- `<button aria-label="通知を開く（未読 12 件）">`。未読数はラベルに必ず含める（視覚バッジだけでは SR に伝わらない）。
- 未読 0 時は `aria-label="通知を開く"`。
- 通知一覧内の各項目は `<li role="listitem">` で種別（リプライ/いいね/フォロー等）をラベル冒頭に含め、色アイコンに依存しない。

### 2.7 検索フィルタ演算子 UI

- 検索ボックスは `role="search"` の親内 `<input type="search" aria-label="投稿と記事を検索" aria-describedby="search-help">`。
- 演算子ヒント（`tag:`, `from:`, `since:` 等）は `<details>` で折りたたみつつ、`aria-describedby` で入力欄に関連付ける。
- 演算子を UI チップで補助する場合、`role="group" aria-label="検索フィルタ"` の内部に押下可能な `<button aria-pressed>` を並べ、色だけでなく押下状態をラベル/`aria-pressed` で示す。
- 結果タブ（すべて/ツイート/記事）は WAI-ARIA Tabs パターン（`role="tablist"` + `role="tab"` + `role="tabpanel"`）、矢印キーで移動。

### 2.8 掲示板（板・スレッド・@メンション）

- 板一覧は `<nav aria-label="掲示板の板一覧">` 内に `<ul>`。
- スレッドリストは `<ol>` + 新着順、各項目には最終レス日時を `<time datetime="…">` で読み上げ可能な形式に。
- 990/1000 レスバナーは `role="status" aria-live="polite"`、1000 達成の停止表示は `role="alert"` でやや強く通知（運用上の重要情報）。
- `@handle` メンションはリンク化し、`aria-label="<表示名>（@handle）のプロフィールへ"` で曖昧さを排除。
- 未ログイン時の CTA「ログインが必要です」は `role="status"` で入力欄を置換せず、**入力欄に `aria-disabled="true"` + 説明テキスト**を併置する（突然の DOM 変化を避ける）。

### 2.9 記事エディタ（Markdown プレビュー切替）

- 編集/プレビュー切替は **Tabs パターン**（`role="tablist"`）で実装し、トグルスイッチにしない（状態認知が容易）。
- 分割プレビュー時はエディタ領域とプレビュー領域にそれぞれ `aria-label="Markdown ソース編集"` / `aria-label="プレビュー（読み上げ用）"` を付ける。SR 利用者はプレビュー側だけ読めばよい設計にする。
- ツールバー（太字/リンク/画像挿入）は `role="toolbar" aria-label="書式ツール"`、各ボタンにショートカット（Ctrl+B 等）を `aria-keyshortcuts` で告知。
- Markdown ソース内の画像挿入時に alt 入力モーダルを出し、未入力のまま確定できないようにする（記事は SEO 対象のため厳しめに）。

### 2.10 モーダル・ダイアログ

- `role="dialog" aria-modal="true" aria-labelledby` を必ず設定。
- **フォーカストラップ**: 開いた直後に先頭フォーカス可能要素へ移動、Tab はダイアログ内を循環、Esc でクローズ、閉じた後は開く前のトリガーへフォーカス復帰。
- shadcn/ui の `Dialog`（Radix ベース）をデフォルトで使用し、独自実装のダイアログ化は禁止。
- 背後のページは `inert` で不活性化し、スクロール抑制を行う。
- 破壊的アクション（アカウント削除、ツイート削除など）の確認ダイアログは、**プライマリボタンをデストラクティブ色**にし、ラベルにも「削除する」と動詞を入れる（「OK」「はい」だけにしない）。

### 2.11 アイコン円形クロップ

- `react-easy-crop` を使う想定だが、マウスドラッグ/ピンチ以外の操作手段を必ず併設する：
  - **キーボード**: 矢印キーで中心点移動、`+`/`-` で拡大/縮小、`R` で回転リセット（`aria-keyshortcuts` で告知）。
  - **スライダー UI**: 拡大率と位置 X/Y をそれぞれ `<input type="range">` で提供し、`aria-valuemin/max/now` + `aria-valuetext` で「拡大率 120%」などを読み上げ。
  - **完了/やり直し/キャンセル** の 3 ボタンをダイアログ下部に明示配置（44x44 以上）。
- プレビュー円形クロップは装飾であり、実画像は `<img alt="アイコンプレビュー">` として提供。

## 3. キーボードナビゲーション設計

### 3.1 グローバルショートカット（X 準拠ベース）

| キー | 動作 | 備考 |
|---|---|---|
| `?` | キーボードショートカットヘルプを開く | モーダル、Esc で閉じる |
| `/` | 検索欄にフォーカス | `input` 内では発火させない |
| `n` | 新規ツイート（コンポーザーを開く） | 未ログイン時は無効 |
| `g` `h` | ホームへ移動 | 2 キー連続（1.5 秒以内） |
| `g` `n` | 通知へ | |
| `g` `m` | DM へ | |
| `g` `p` | 自分のプロフィールへ | |
| `g` `s` | 設定へ | |
| `j` / `k` | TL 内で次/前のツイートへフォーカス | |
| `l` | フォーカス中ツイートへリアクション（既定 like） | |
| `r` | 返信 | |
| `t` | リツイート確認 | |
| `Esc` | モーダル/メニューを閉じる、新規入力を破棄確認 | |

- **ユーザーが無効化できる設定** を `/settings` に用意（運動機能障害や入力支援ソフト利用者向け）。
- `input` / `textarea` / `contenteditable` フォーカス中はグローバルショートカットを無効化。
- ショートカットは `aria-keyshortcuts` で明示、ヘルプダイアログ（`?`）で一覧を提供。

### 3.2 Tab 順序設計

原則として **画面上の視覚順序 = DOM 順序 = Tab 順序** を維持する。`tabindex > 0` は使用禁止。

推奨 Tab フロー（ホーム画面の例）:

1. Skip Link「メインコンテンツへスキップ」 → `<main>` にフォーカス
2. グローバルナビ（左サイドバー: ホーム → 通知 → DM → 検索 → 掲示板 → 記事 → お気に入り → プロフィール → 設定）
3. ツイート投稿 FAB（スマホではボトムナビ右）
4. メイン TL（ツイート 1 件目から順に、各ツイート内はリンク → アクションバーの順）
5. 右サイドバー（トレンドタグ → おすすめユーザー）

スマホではボトムナビが最上位 landmark `role="navigation"` として Tab 導線に含まれる。

### 3.3 フォーカスリング

- **SC 2.4.11 Focus Appearance**（WCAG 2.2 新設）に準拠。
- Tailwind トークン:
  ```css
  @layer utilities {
    .focus-ring {
      @apply outline-none
             ring-2 ring-offset-2
             ring-[color:var(--color-focus)]
             ring-offset-[color:var(--color-surface)];
    }
  }
  ```
- `--color-focus` はライト `oklch(62% 0.19 250)`、ダーク `oklch(78% 0.18 250)` を基準とし、背景に対して **3:1 以上** のコントラストを確保。
- `:focus-visible` のみで適用し、マウスクリック時のリングは抑制（ただし抑制によって完全に見えなくならないようにする）。
- shadcn/ui 既定のフォーカス表現では弱い箇所があるため、`components.json` レベルで上書きする。

## 4. カラーコントラスト

### 4.1 最低比率

| 対象 | 最低比率 | 参照 SC |
|---|---|---|
| 本文テキスト | 4.5:1 | 1.4.3 |
| 18pt 以上 / 14pt bold 以上 | 3:1 | 1.4.3 |
| アイコン・UI 境界・フォーカスリング | 3:1 | 1.4.11 |
| プレースホルダテキスト | 4.5:1 | 1.4.3（扱い上テキスト） |
| リアクション絵文字 + ラベル | 4.5:1（ラベル部） | 1.4.3 |

### 4.2 ライト/ダーク両対応

- デザイントークンは `oklch` で管理し、ライト/ダーク両モードで **同一の意味ロール** に対応するペアを用意（例: `--color-text`, `--color-text-muted`, `--color-accent`）。
- ダークモードの純黒 (#000) は避け、`oklch(14% 0 0)` 程度を基準とする（ハレーションを抑え、低視覚者の読みやすさを改善）。
- プレミアムバッジ・Bot マーク・記事マーク・リアクションは **色 + 形状 + ラベル** の 3 冗長化で情報を伝え、**色だけに依存しない**（SC 1.4.1）。

### 4.3 禁則

- リアクション数の増減を「赤字/緑字だけ」で示すことを禁止（矢印アイコン + テキストを併用）。
- リンクを色だけで示すことを禁止（下線 or ホバー時以外にも視覚的差異を持たせる）。
- コードブロックのシンタックスハイライトに依存した意味伝達（色だけで変数と予約語を区別する等）を本文情報として扱わない。

## 5. スクリーンリーダー対応

### 5.1 ランドマークと見出し構造

- ページ 1 枚ごとに必ず 1 つの `<main>` と 1 つの `<h1>`。
- `<header>` / `<nav>` / `<main>` / `<aside>` / `<footer>` を使い、`role="banner"` / `role="navigation"` / `role="main"` / `role="complementary"` / `role="contentinfo"` を暗黙的に得る。
- 見出しレベルは **スキップしない**（h1 → h3 は禁止）。
- ページごとの `<h1>` 例:
  - ホーム: 「ホームタイムライン」
  - ツイート詳細: 「<表示名> のツイート」
  - プロフィール: 「<表示名>（@handle）のプロフィール」
  - 記事: 記事タイトルをそのまま
  - 掲示板板詳細: 「<板名>」
  - スレッド詳細: スレッドタイトル

### 5.2 aria-live の使い分け

| 用途 | 設定 |
|---|---|
| 文字数カウント（通常域） | `aria-live="polite"` + 閾値ベース更新 |
| 文字数オーバー / 送信エラー | `role="alert"`（assertive 扱い） |
| 新着ツイート到着通知（ボタン出現） | `aria-live="polite"` |
| TL ローディング | `role="status" aria-live="polite"` |
| DM 新着メッセージ | `aria-live="polite"` + `aria-relevant="additions"` |
| DM タイピング中 | **読み上げない**（視覚のみ）／3 秒超で 1 回だけ `role="status"` |
| 通報送信成功 | `role="status"` |
| セッション失効 | `role="alert"` |

- `aria-live="assertive"` は**本当に即時中断が必要な場合のみ**。乱用は SR 利用者にとって著しくうるさく、体験を損なう。

### 5.3 WAI-ARIA パターン採用表

| コンポーネント | パターン | 実装基盤 |
|---|---|---|
| ダイアログ | Modal Dialog | shadcn `Dialog`（Radix） |
| メニュー（ツイート「…」） | Menu Button | shadcn `DropdownMenu` |
| リアクションメニュー | Menu with `menuitemradio` | Radix Menu + 独自拡張 |
| 検索結果タブ | Tabs | shadcn `Tabs`（Radix） |
| タグコンボボックス | Combobox（Editable, with listbox popup） | Radix Combobox or cmdk |
| トースト通知 | Status Message | shadcn `Toast`（Radix） |
| ツールチップ | Tooltip | shadcn `Tooltip`（Radix）、キーボードフォーカス時にも表示 |
| アコーディオン（演算子ヘルプ） | Accordion | shadcn `Accordion` |
| タイムライン | Feed pattern | 独自、`role="feed"` 準拠 |
| トグル（設定 ON/OFF） | Switch | shadcn `Switch`、`role="switch" aria-checked` |

原則として **Radix/shadcn に実装がある場合は独自実装を禁止**。どうしても必要なら WAI-ARIA Authoring Practices に準拠する。

## 6. モバイル a11y

### 6.1 タッチターゲット

- **SC 2.5.8 Target Size (Minimum)**: 24x24 CSS px 以上（WCAG 2.2 新設）。
- 本プロダクトでは **モバイルは 44x44 px（iOS HIG）、Web 全般で 24x24 px を絶対下限** とする。ツイートカード内アクションバー（リプライ/RT/リアクション/共有）は **44x44 px** を採用。
- ターゲット間は **最低 4 px のスペーシング**（Tailwind `gap-1` 以上）を確保。

### 6.2 ピンチズーム

- `<meta name="viewport" content="width=device-width, initial-scale=1">` のみとし、`maximum-scale` / `user-scalable=no` を **絶対に設定しない**（SC 1.4.4 / 1.4.10）。
- 400% ズーム時に水平スクロールが発生しないよう、Tailwind の固定幅（`w-*`）より `max-w-*` / `clamp()` を優先して使う。

### 6.3 モバイルナビ一貫性

- ボトムナビ 5 タブ（ホーム / 検索 / 投稿 / 通知 / DM）は全画面で同一位置・同一順序を維持（SC 3.2.3 Consistent Navigation）。
- セーフエリア `env(safe-area-inset-bottom)` を考慮し、iOS ホームインジケータと被らない。
- ハンバーガーメニュー開閉は `aria-expanded` / `aria-controls` + フォーカストラップで実装。
- デバイスの **戻るジェスチャ・戻るボタン** でモーダルや右ドロワーが閉じること（history API を連動）。

### 6.4 方向固定の禁止

- SC 1.3.4: 縦横どちらの画面向きでも使用可能に保つ。
- カメラ・クロップ UI など本質的に向き依存の画面のみ例外とし、その旨を明示する。

## 7. 禁止事項・注意事項

以下は **リリース前チェックリストで必ずブロック** する項目である。

- [ ] **音声/動画の autoplay**（音付きは絶対禁止、無音でも `prefers-reduced-motion` 尊重。MVP は動画なしだが OGP 内の外部動画埋込も含めて禁止）
- [ ] **キーボードトラップ**（モーダル以外で Tab が戻れなくなる UI。モーダルは意図的トラップで可）
- [ ] **`<blink>` / `<marquee>` / 等価 CSS アニメーション**
- [ ] **1 秒 3 回以上の点滅**（SC 2.3.1、脳への影響・光感受性発作対策）
- [ ] **色のみで情報を伝える UI**（エラー色赤のみ、成功色緑のみ、等）
- [ ] **`outline: none` のみで代替なし**（フォーカス完全消失）
- [ ] **`title` 属性だけで代替テキスト代用**（SR/モバイルで読まれない）
- [ ] **`role="button"` を div に付けただけで `tabindex`/`keydown` 未実装**
- [ ] **無限読み込みでキーボードフォーカスがロスト**
- [ ] **同じ入力を 2 回要求**（SC 3.3.7 Redundant Entry。サインアップ後のプロフィール初期設定で既に取得済の情報を再入力させない）
- [ ] **アクセシブル認証で CAPTCHA を唯一の手段にする**（SC 3.3.8、代替手段を用意）

## 8. CI/CD 組み込み

### 8.1 axe-core + Playwright

`@axe-core/playwright` を Playwright の共通 fixture から呼び出し、公開ページと主要画面で走査する。

```ts
// e2e/a11y.spec.ts
import AxeBuilder from '@axe-core/playwright';
import { test, expect } from '@playwright/test';

const publicRoutes = [
  '/explore',
  '/tweet/seed-001',
  '/u/seed_user',
  '/tag/typescript',
  '/boards',
  '/threads/seed-001',
  '/articles/seed-first-article',
];

for (const path of publicRoutes) {
  test(`a11y: ${path}`, async ({ page }) => {
    await page.goto(path);
    await page.waitForLoadState('networkidle');
    const results = await new AxeBuilder({ page })
      .withTags(['wcag2a', 'wcag2aa', 'wcag22aa', 'best-practice'])
      .disableRules(['color-contrast-enhanced']) // AAA は対象外
      .analyze();

    const blocking = results.violations.filter(v =>
      v.impact === 'serious' || v.impact === 'critical'
    );
    expect(blocking, JSON.stringify(blocking, null, 2)).toEqual([]);
  });
}
```

- `serious` / `critical` が 1 件でもあれば CI レッド、`moderate` / `minor` はレポートのみ。
- 誤検知が出たルールは `.axe-config.json` で明示的に除外し、**理由をコメント付き PR で管理**（サイレント無効化禁止）。

### 8.2 Lighthouse CI

`.lighthouserc.json` 例:

```json
{
  "ci": {
    "collect": { "staticDistDir": "./out", "numberOfRuns": 3 },
    "assert": {
      "assertions": {
        "categories:accessibility": ["error", { "minScore": 0.95 }],
        "categories:performance": ["warn", { "minScore": 0.85 }],
        "categories:best-practices": ["warn", { "minScore": 0.9 }]
      }
    }
  }
}
```

- GitHub Actions で PR ごとにホーム（要認証 stub）/ ツイート詳細 / プロフィール / 記事 / 板 / スレッドの 6 ルートを走査。
- a11y スコア 95 未満は**マージブロック**。

### 8.3 PR テンプレート追記

`.github/pull_request_template.md` に a11y チェックリストを追加：

```markdown
## アクセシビリティ
- [ ] キーボードのみで主要フローを操作した
- [ ] `:focus-visible` でフォーカスリングが明確に見える
- [ ] 追加したボタン/アイコン/画像に `aria-label` または `alt` を設定した
- [ ] 追加した動的領域の `aria-live` 方針を決めた（polite/assertive/none）
- [ ] 色以外に情報の手がかりがある
- [ ] 44x44（モバイル）/ 24x24（Web）のタップ領域を満たしている
- [ ] axe-core CI がグリーン / Lighthouse a11y >= 95
```

### 8.4 Storybook での a11y アドオン

- `@storybook/addon-a11y` を導入し、shadcn/ui ベースの個別コンポーネントを**そのストーリー単位で axe 走査**。ビジュアル QA と同時に a11y もレビューする。
